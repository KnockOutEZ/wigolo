# How to properly handle async/await errors in JavaScript?

**Asked:** 2 years ago | **Modified:** 3 months ago | **Viewed:** 189k times

**Tags:** javascript, async-await, error-handling, promise, try-catch

---

## Question

**Score: 156**

I have multiple async functions that can throw errors. I want to handle errors gracefully without wrapping every single call in try/catch. What are the best patterns for async error handling?

```javascript
async function getUser(id) {
  const response = await fetch(`/api/users/${id}`);
  if (!response.ok) throw new Error(`User not found: ${id}`);
  return response.json();
}

async function getUserPosts(userId) {
  const response = await fetch(`/api/users/${userId}/posts`);
  if (!response.ok) throw new Error(`Posts not found for: ${userId}`);
  return response.json();
}

async function main() {
  // This gets messy with individual try/catch blocks
  try {
    const user = await getUser(1);
    try {
      const posts = await getUserPosts(user.id);
      try {
        // ... more nested operations
      } catch (e) { /* ... */ }
    } catch (e) { /* ... */ }
  } catch (e) { /* ... */ }
}
```

## Accepted Answer

**Score: 234** | Answered by **async_master**

There are several clean patterns. Here are the most practical ones:

### Pattern 1: Go-style Error Tuples

Create a wrapper that returns `[error, data]` tuples:

```javascript
function to(promise) {
  return promise
    .then(data => [null, data])
    .catch(err => [err, null]);
}

async function main() {
  const [userErr, user] = await to(getUser(1));
  if (userErr) {
    console.error('Failed to get user:', userErr.message);
    return;
  }

  const [postsErr, posts] = await to(getUserPosts(user.id));
  if (postsErr) {
    console.error('Failed to get posts:', postsErr.message);
    return;
  }

  console.log(`${user.name} has ${posts.length} posts`);
}
```

### Pattern 2: Single try/catch with Error Types

Use custom error classes to differentiate in a single catch block:

```javascript
class UserNotFoundError extends Error {
  constructor(id) {
    super(`User not found: ${id}`);
    this.name = 'UserNotFoundError';
    this.userId = id;
  }
}

class PostsNotFoundError extends Error {
  constructor(userId) {
    super(`Posts not found for user: ${userId}`);
    this.name = 'PostsNotFoundError';
    this.userId = userId;
  }
}

async function main() {
  try {
    const user = await getUser(1);
    const posts = await getUserPosts(user.id);
    console.log(`${user.name} has ${posts.length} posts`);
  } catch (err) {
    if (err instanceof UserNotFoundError) {
      console.error(`User ${err.userId} does not exist`);
    } else if (err instanceof PostsNotFoundError) {
      console.error(`No posts for user ${err.userId}`);
    } else {
      console.error('Unexpected error:', err);
      throw err; // re-throw unknown errors
    }
  }
}
```

### Pattern 3: Promise.allSettled for Independent Operations

When operations do not depend on each other:

```javascript
async function loadDashboard(userId) {
  const results = await Promise.allSettled([
    getUser(userId),
    getUserPosts(userId),
    getUserNotifications(userId),
    getUserSettings(userId),
  ]);

  const [userResult, postsResult, notifResult, settingsResult] = results;

  return {
    user: userResult.status === 'fulfilled' ? userResult.value : null,
    posts: postsResult.status === 'fulfilled' ? postsResult.value : [],
    notifications: notifResult.status === 'fulfilled' ? notifResult.value : [],
    settings: settingsResult.status === 'fulfilled' ? settingsResult.value : defaults,
  };
}
```

### Pattern 4: Result Type (TypeScript)

For TypeScript users, a discriminated union Result type:

```typescript
type Result<T, E = Error> =
  | { ok: true; value: T }
  | { ok: false; error: E };

async function safeGetUser(id: string): Promise<Result<User>> {
  try {
    const user = await getUser(id);
    return { ok: true, value: user };
  } catch (error) {
    return { ok: false, error: error as Error };
  }
}

async function main() {
  const result = await safeGetUser('1');
  if (!result.ok) {
    console.error(result.error.message);
    return;
  }
  // result.value is typed as User here
  console.log(result.value.name);
}
```

### Which Pattern to Use?

| Pattern | Best For | Trade-offs |
|---------|----------|------------|
| Go-style tuples | Many sequential async calls | Verbose null checks |
| Custom errors + single catch | Operations that logically belong together | Need to define error classes |
| Promise.allSettled | Independent parallel operations | Cannot handle sequential dependencies |
| Result type | TypeScript projects needing exhaustive error handling | More boilerplate per function |

## Other Answers

### Answer 2

**Score: 67** | Answered by **functional_dev**

You can also use a higher-order function for consistent error handling across route handlers (Express/Koa):

```javascript
const asyncHandler = (fn) => (req, res, next) => {
  Promise.resolve(fn(req, res, next)).catch(next);
};

// Usage
app.get('/users/:id', asyncHandler(async (req, res) => {
  const user = await getUser(req.params.id);
  res.json(user);
}));
```

This catches any thrown error and passes it to Express's error middleware.

### Answer 3

**Score: 31** | Answered by **node_ninja**

For Node.js specifically, you should also handle unhandled rejections globally as a safety net:

```javascript
process.on('unhandledRejection', (reason, promise) => {
  console.error('Unhandled Rejection at:', promise, 'reason:', reason);
  // Log to monitoring service
  // Optionally exit: process.exit(1);
});
```

This should never be your primary error handling strategy, but it catches anything that slips through.
